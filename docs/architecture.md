# Architecture

Notorium keeps framework entrypoints thin and pushes product logic into feature modules.

Audience: agents and contributors making structural, cross-boundary, or feature-module changes.

Related docs: [Docs Index](./README.md), [UI Conventions](./ui-conventions.md), [Agent Guide](../AGENTS.md), [README](../README.md)

## Source Layout

```text
src/
|-- app/                 Next.js App Router pages, layouts, API routes, and Server Actions
|   |-- (app)/           Authenticated pages
|   `-- actions/         Client-callable server boundary
|-- components/          Feature UI, shared UI, the left menu (components/navbar), auth forms, and shadcn primitives
|-- features/            Feature queries, mutations, validation, mappers, and domain helpers
|   `-- planning/        Planning route helpers and planning-specific data shaping
|-- db/                  Drizzle schema and database client
|-- lib/                 Auth, server contracts, dates, editor, AI, email, rate limits, storage
`-- env.ts               Runtime environment validation
```

## Data Flow

```text
Server Component -> src/features/*/queries.ts -> Drizzle -> PostgreSQL
Client Component -> src/app/actions/* -> validation/auth -> src/features/*/mutations.ts -> Drizzle
Scheduled Workflow -> src/app/api/* -> auth/config checks -> src/features/* service functions
```

## Ownership Boundaries

- `src/app/` owns framework routing, layouts, pages, API routes, and thin Server Actions.
- `src/components/` owns reusable and feature UI; shadcn primitives stay in `src/components/ui/`.
- `src/features/` owns product read/write logic, validation, mapping, constants, and feature-local types.
- Dense algorithms may live in feature subfolders, such as `src/features/flashcards/fsrs/`.
- Flashcard refine (streak detection, similar-card discovery, relate/merge mutations) lives in `src/features/flashcards/refine/`, with its Server Actions in `src/app/actions/flashcards-refine.ts`. The UI is a mode of the manage view in `src/components/flashcards/manage/` (refine results table, merge/improve dialogs, `use-refine-card-actions.ts`), entered from the manage toolbar AI dropdown. Merges snapshot source cards into the `flashcard_merge_log` table so lineage survives deletion.
- Cloze flashcards: parsing and per-ordinal front/back rendering of `{{cN::answer}}` sources live in `src/features/flashcards/cloze.ts`; the fan-out/sibling-sync/note-delete write logic lives in `src/features/flashcards/cloze-mutations.ts` (delegated to from `mutations.ts`). The `flashcard` row stays the scheduled unit: each deletion ordinal is its own row sharing a `clozeNoteId`, with `type`/`clozeOrdinal`/`clozeSource` columns. Front/back are pre-rendered at write time so review, search, and the manage table read them unchanged. Bulk mutations were split into `src/features/flashcards/bulk-mutations.ts` to keep `mutations.ts` focused.
- Image occlusion flashcards follow the same sibling pattern as cloze. Region geometry and derived-content helpers live in `src/features/flashcards/occlusion.ts` (pure, also imported by `db/schema.ts` for column typing and by client components for pixel↔normalized projection); the fan-out/sibling-sync/note-delete write logic plus source-image blob cleanup live in `src/features/flashcards/occlusion-mutations.ts` (delegated to from `mutations.ts`). Each mask region is its own row sharing an `occlusionNoteId`, with `occlusionMaskId`/`occlusionImagePathname`/`occlusionRegions` (jsonb, normalized 0..1) columns; `front`/`frontNormalized`/`back` hold derived text so search and the manage table read them unchanged. The source image uploads through the shared `POST /api/attachments/image` route handler (returns the blob pathname the card persists). Authoring UI (`react-rnd` drag/resize masks) and the review/preview renderer live in `src/components/flashcards/occlusion/`.
- Planning-specific URL and view helpers live in `src/features/planning/`.
- Media uploads (note/flashcard/mindmap editor images, occlusion source images, assessment attachments) funnel through `src/features/attachments/mutations.ts`, then run `src/lib/media-storage/optimize-image.ts` before handing bytes to the storage provider (`src/lib/media-storage/provider.ts` → `vercel-blob-provider.ts`). The server upload path is therefore: validate → optimize → store. Image uploads (the editor and occlusion paths) transfer raw multipart bytes through the `POST /api/attachments/image` route handler into `uploadAttachmentImageForUser`, not a Server Action: Server Actions cap request bodies at 1 MB (below Vercel's 4.5 MB function-body ceiling, so it cannot just be raised), and multipart avoids base64's ~33% inflation. The shared client helper is `src/lib/attachments/upload-attachment-image.ts`. This route handler is the one sanctioned client-callable write endpoint outside Server Actions. Assessment attachments still travel base64 through the `uploadAssessmentAttachment` Server Action (they record DB metadata and allow non-image files), decoding via `src/lib/media-storage/decode-base64.ts`. `optimizeImageForStorage` re-encodes raster images to WebP with an EXIF-orientation fix and a max-dimension cap (sharp, `serverExternalPackages`), and falls back to the original bytes when sharp cannot decode them or WebP would not be smaller, so it never enlarges or corrupts an upload. Non-image uploads (PDFs, docs) pass through untouched; the resulting `mimeType`/`sizeBytes`/`fileName` from optimization are what assessment attachments persist. Library PDF books upload client→Blob directly (`@vercel/blob/client`) and are not part of this server-side optimize path.
- Library (`src/features/library/`) is a top-level, `userId`-scoped feature for uploading PDF books and tracking reading position. Books store file metadata plus `currentPage`/`totalPages` on a single `library_book` row (one user : one book : one progress, so no separate progress table). Uploads go browser→Blob directly: `generateLibraryUploadToken` (`src/app/actions/library.ts`) mints a scoped client token via `generateClientTokenFromReadWriteToken` from `@vercel/blob/client` (locking the pathname, `maximumSizeInBytes`, and `application/pdf` content type) after checking the per-user book limit, the client uploads the PDF straight to Blob with `upload()` from `add-book-dialog.tsx`, and the `uploadBook` Server Action then only records the `library_book` metadata row (it never carries the file bytes). This bypasses both the 1 MB Server Action body cap and Vercel's 4.5 MB function-body ceiling, so books can be far larger than image attachments (`libraryBookMaxBytes`). The PDF is served back through the ownership-checked route `src/app/api/library/[id]/route.ts` (inline `content-disposition`). The reader is built on EmbedPDF (`@embedpdf/*`), whose pdfium WebAssembly engine renders pages and powers the zoom/spread/fullscreen/thumbnail/bookmark plugins, with the selection plugin adding a selectable text layer over each page. Because the engine is browser-only WASM, `book-reader.tsx` loads the implementation through `next/dynamic` with `ssr: false`; `book-reader-loading-frame.tsx` preserves the reader toolbar/sidebar/canvas geometry while that dynamic bundle, the pdfium engine, and the PDF document load; `book-reader-surface.tsx` initializes the engine (`usePdfiumEngine`) and registers plugins inside the `EmbedPDF` provider, `book-reader-layout.tsx` composes the toolbar, sidebar, and scrolling viewport, and `book-reader-toolbar.tsx`/`book-reader-sidebar.tsx` drive the controls. The sidebar's Pages/Content/Highlights switch renders `book-reader-thumbnails.tsx` (the virtualized page rail), `book-reader-outline.tsx` (the bookmark plugin's table of contents), or `book-reader-highlights.tsx` (the live annotation-plugin highlight list); LINK annotations jump through the shared `use-reader-navigate-target.ts` hook, which drives the annotation plugin's navigator (smooth scroll, precise in-page position) and records reader back-history; outline entries instead scroll to the destination page directly with `behavior: "instant"` (the navigator has no instant option) and fall back to that shared hook only for external URI targets. Highlight entries scroll directly to the annotation rectangle with `behavior: "instant"`, record back-history, and select the annotation through the annotation plugin so the existing highlight note/delete panel owns edits. `use-reading-position.ts` owns the page-save logic, reading the scroll plugin's `currentPage` (debounced + deduped saves via `updateReadingPage`, restore-before-save guard, and a visibilitychange/unmount flush). `updateReadingPage` deliberately does not `revalidatePath` since it fires on every scroll. The pdfium WASM loads from a version-pinned jsDelivr CDN by default (override via `usePdfiumEngine({ wasmUrl })`). Because PDFs upload client→Blob directly (above), no `serverActions.bodySizeLimit` override is needed for the library.
- Reader highlights live in their own feature (`src/features/library-annotations/`), kept separate from `src/features/library/` so the read-only book metadata path and the per-highlight write path stay decoupled. The PDF blob is never modified: each highlight is one `library_annotation` row storing the EmbedPDF `PdfAnnotationObject` verbatim as `jsonb` (its `contents` field carries the note), keyed by `(bookId, annotationUid)` for idempotent upserts. The page loads a book's highlights server-side (`getAnnotationsForBook`) and threads them as `initialAnnotations` into the reader; `book-reader-annotations.ts` imports them into the engine once on open, then subscribes to the annotation plugin's `onAnnotationEvent` to mirror committed create/update/delete back through the `saveBookAnnotation`/`deleteBookAnnotation` Server Actions (no `revalidatePath`, mirroring `updateReadingPage`). Dates round-trip as ISO strings in the JSON column and are revived to `Date` by the feature mappers. The annotation plugin is registered with `LockModeType.Exclude` over a custom `userHighlight` category (see `book-reader-annotation-config.ts`): every existing PDF annotation and LINK stays read-only/clickable while only user highlights are selectable, so `book-reader-annotation-menu.tsx` (the per-highlight note + delete panel passed to `AnnotationLayer`'s `selectionMenu`) and the Highlighter toggle in `book-reader-interaction-tools.tsx` act only on user highlights. The built-in `highlight` tool is overridden to tag new highlights with that category and paint them in the reader's highlighter color; imported Notorium highlights are also tagged before they enter EmbedPDF so `book-reader-highlights.tsx` can list only user-owned highlights rather than every PDF-native highlight annotation.
- Notes and mindmaps are both subject-scoped and surfaced together in a subject's "Documents" area under `src/app/(app)/subjects/[id]/documents/` (`documents/notes/[noteId]/`, `documents/mindmaps/[mindmapId]/`). Document navigation comes from the global subject tree sidebar. The subject page is academic-only: a full-path breadcrumb plus the attendance and assessment summaries (`src/components/subjects/subject-detail-content.tsx`) — the documents, flashcards, and subfolders that used to live here are all managed in the sidebar instead. General subjects have no dashboard, so they are never a navigation target. `getSubjectPageHref` (`src/features/subjects/constants.ts`) is the single source of that rule — it returns a page href for academic subjects and `null` for general ones — and every surface that could link a subject routes through it: the sidebar tree label is inert text (expand via the chevron only), breadcrumb segments for general ancestors/subjects render unlinked (subject page and note/mindmap detail), the home "recent subjects" tiles render general subjects non-clickable, document deletes for a general subject return home, and `getSearchDataForUser` omits general subjects from subject results (their notes/mindmaps stay searchable). A direct URL visit to a general subject shows a short "managed in the sidebar" notice (`GeneralSubjectNotice`), which the sidebar's active-path auto-expand then reveals in the tree. `src/features/documents/` is a thin read layer that merges `src/features/notes` and `src/features/mindmaps` results into one recency-ordered list (`DocumentListItem`) used by the sidebar and the home dashboard. Note and mindmap detail pages do not render an in-page sidebar — document navigation comes from the global subject tree sidebar — so their editor/canvas spans the full content width. The per-document kebab (Edit/Delete/Generate/Copy/Export) is shared via `src/components/documents/document-row-menu.tsx` (the menu) and `use-document-row-dialogs.tsx` (a hook owning the dialog state + note copy, returning the row handlers and the dialog JSX); the tree sidebar (`src/components/subjects/tree/`) consumes it. Because the tree lives in the app shell, the authenticated layout (`src/app/(app)/layout.tsx`) loads subject options once and threads them with `aiEnabled` down to the tree so the "Generate flashcards" action works from any page.
- Mindmaps (`src/features/mindmaps/`) are scoped by `userId` and `subjectId` (mirroring notes). The free-form canvas uses React Flow (`@xyflow/react`), loaded client-side, and persists its node/edge graph as JSON in the `mindmap.data` column.
- The authenticated shell (`src/components/shared/app-layout-client.tsx`) renders the collapsible left menu (`src/components/navbar/`) beside the page content; there is no top navbar. Pages render a sticky breadcrumb top bar via `src/components/shared/page-top-bar.tsx` (`PageTopBar` and the matching `PageTopBarSkeleton` for `loading.tsx`), which is the location indicator and replaces per-page back links. Card/detail pages compose a sticky breadcrumb plus centered body through the shared `DetailPageLayout` (`src/components/shared/detail-page-layout.tsx`), which wraps `PageTopBar` over an `AppPageContainer` (default `max-w-5xl`, narrower per page via `maxWidth`: e.g. flashcard/admin `3xl`, assessment `4xl`, subject `5xl`); document workspaces (documents list, note, mindmap) stay full-width (`max-w-7xl`) and keep their own layout for zen/embedded modes. The root route (`src/app/(app)/page.tsx`) is the Home dashboard rendered inside the authenticated shell; it self-gates with `requireSession()` (signed-out users go to `/login`) and composes overview cards from existing feature queries (flashcards due, upcoming assessments, recent documents across subjects, subjects grid).
- Floating windows (`src/components/windows/`) are a shell-level client window manager that overlays editors on top of the current page without changing the route, so an immersive page such as the book reader stays mounted behind. `WindowManagerProvider` (mounted in `app-layout-client.tsx`) holds the open-window stack and the single `activeWindowId`; `WindowOverlay` portals every window to `document.body` and keeps inactive ones mounted-but-hidden so their state survives minimize/restore, while `WindowDock` is the bottom taskbar. `WindowContent` resolves each window: mindmap/note panes load their document client-side via the existing `getMindmapById`/`getNoteById` Server Actions and render `MindmapDetail`/`NoteDetail` in `embedded` mode (no `PageTopBar`, fill height, delete closes the window instead of navigating); flashcard windows with no `docId` reuse the create form through `useCreateFlashcardForm` + `CreateFlashcardFormPanel`, while flashcard windows with a `docId` load the card through `getFlashcardForManage` and reuse `EditFlashcardDialog` in no-dialog mode so submit/AI/split/delete behavior stays shared. Windows open only from the command palette's "Windows" group; "Open Document in Window" is backed by the `getOpenableDocuments` Server Action (`src/app/actions/documents.ts`, a thin wrapper over `getRecentDocumentsForUser`), and "Edit Flashcard in Window" reuses `getFlashcardsManagePage` for the picker search. The close buttons (title bar and dock chip) call `requestCloseWindow`, which runs a per-window guarded-close handler that each embedded editor registers via `useWindowCloseGuard` (`src/lib/editor/`): flashcard panels register their discard-dialog flow, while `MindmapDetail`/`NoteDetail` register an autosave flush. A window with no registered guard closes directly.
- `src/db/` owns the Drizzle schema, relations, and database client.
- `src/lib/` owns cross-cutting project helpers and wrappers for external services.
- `src/env.ts` owns runtime environment validation.

Cross-cutting services live behind project-owned helpers in `src/lib/`, including Better Auth, AI providers, email, media storage, and rate limiting.

The rich-text editor and read-only renderer (`src/components/shared/tiptap-editor.tsx` and `tiptap-renderer.tsx`) share their Tiptap extension sets through `src/lib/editor/build*Extensions()` helpers (tables, math). Both surfaces must register the same extensions: Tiptap drops unknown nodes when parsing stored HTML, so math or tables present only in the editor would vanish in the renderer. Math nodes (KaTeX) serialize their LaTeX into a `data-latex` attribute, which `src/lib/editor/rich-text.ts` surfaces as text so equations stay searchable and within length limits.

## Enforced Boundaries

Two boundaries above are machine-checked, not just convention (see the Automated Guards section in [AGENTS.md](../AGENTS.md)):

- Server Actions (`src/app/actions/*`) cannot import `@/db/index`; they go through `src/features/*`. Enforced by a `noRestrictedImports` override in `biome.json`.
- Components (`src/components/*`) cannot import `@/features/*/mutations`; writes flow through Server Actions. Enforced by the same rule. Server Components reading from `queries.ts` live under `src/app/*`, so they are unaffected.

A separate guard, `bun run check:size`, keeps files under 500 lines.
